# 第10章仕様書：動的攻防モード

## 1. 章の位置づけ

第10章は、第1章から第9章まで続いた「伴走型の学習ステージ」を終了し、プレイヤーが攻撃側、ゲーム側が防御側として振る舞う **動的攻防モード** へ移行する章である。

第1章〜第4章では、順序、状態、数値、複合戦略という個別メカニクスを学んだ。第5章〜第7章では、自由実験、自動化、戦略探索へ進んだ。第8章〜第9章では、資源制約、防御閾値、適応防御との読み合いを導入した。

第10章では、これらを統合し、固定された正解ルートを探すのではなく、プレイヤーが自分で攻撃計画を立て、ゲーム側の防御反応を観測しながら、複数ラウンドで攻略を進める。

ただし、本章の対象は **ゲーム内の閉じた仮想DEX / 仮想取引基盤 / 仮想防御システム** のみであり、現実のネットワーク、外部サービス、外部API、実在システムに対する攻撃や侵入を扱わない。

---

## 2. 第10章の目的

### 2.1 教育目的
プレイヤーに、実際の安全性検証では単一の正解をなぞるのではなく、観測、仮説、実行、反応、修正という反復的なプロセスが重要であると理解させる。

### 2.2 体験目的
プレイヤーが、以下の体験を得られるようにする。

- 固定ステージではなく、動的に変化する防御を相手にする
- 自分で攻撃計画を構築する
- 防御のログや反応を読み、次の手を変える
- 成功、失敗、部分成功をもとに戦略を改善する
- 最終的に「自分の戦略で突破した」感覚を得る

### 2.3 システム理解としての目的
システムの防御は静的なルールだけでなく、観測、制限、隔離、再設定、スコア調整などの応答を持ちうることを理解させる。

---

## 3. 安全スコープ

第10章では、攻撃側・防御側という表現を使うが、すべてゲーム内シミュレーションとして扱う。

### 3.1 許可する対象
- ゲーム内仮想DEX
- ゲーム内仮想注文処理システム
- ゲーム内仮想マッチングエンジン
- ゲーム内仮想防御エージェント
- ゲーム内仮想資源モデル

### 3.2 禁止する対象
- 実在Webサイト
- 実在API
- 実ネットワーク
- 実サーバ
- 実アカウント
- 実サービスへの負荷付与
- 実システムへの侵入や侵害

### 3.3 実装上の制約
- 外部ネットワーク送信は行わない
- 任意コード実行は許可しない
- スクリプトは専用DSLとして解釈する
- 実行対象はゲーム内サンドボックスに限定する
- リソース消費はゲーム内数値として扱う

## 4. 第10章の中心テーマ

第10章は次のように定義する。

**「伴走型ステージを終え、プレイヤーが攻撃計画を立て、ゲーム側の動的防御を相手に複数ラウンドで攻略する章」**

この章では、プレイヤーは以下を行う。

- 攻撃目標を確認する
- 戦略を選ぶ、組む、またはDSLで記述する
- 実行する
- 防御反応を観測する
- 戦略を更新する
- キャンペーンスコアを高める

---

## 5. 第10章の中核概念

### 5.1 伴走終了
第10章以降では、ゲーム側が逐次的に「次はこれをしてください」と強く誘導しない。プレイヤーは自分で方針を立てる。

### 5.2 動的防御
ゲーム側は、固定条件だけでなく、プレイヤーの履歴、戦略傾向、負荷傾向、成功パターンを見て防御状態を変化させる。

### 5.3 複数ラウンド
1回の成功や失敗で終わらず、複数ラウンドを通じてスコアや防御状態が変化する。

### 5.4 観測と仮説
プレイヤーは防御ログ、検知スコア、遮断理由、リソース状態を観測し、次の戦略を考える。

### 5.5 スコアリング
単なる突破可否ではなく、利得、成功率、低検知性、資源効率、安定性などを含めて評価する。

## 6. 第10章の章構成案

第10章は4ステージ構成を基本とする。

1. 導入：動的攻防モードのルールを理解する
2. 初回ラウンド：自分の攻撃計画を作って実行する
3. 防御読解：防御反応を読み、戦略を更新する
4. キャンペーン：複数ラウンドで総合スコアを競う

### 第10章の流れ
- **10-1 ここからは対戦モード**：伴走終了と動的防御のルールを理解する
- **10-2 自分の初回攻撃計画を組め**：攻撃計画を組み、最初のラウンドを実行する
- **10-3 防御を読んで戦略を変えろ**：防御反応を分析し、戦略を更新する
- **10-4 動的防御キャンペーンを突破せよ**：複数ラウンドで総合スコアを高める

### Stage 10-1: ここからは対戦モード

**学習目標**

**ステージの役割**
伴走フェーズから動的攻防フェーズへの切り替えを明確にする。

**初期状態**
- プレイヤーには第1〜9章で学んだ要素のサマリが提示される

- 攻撃計画は未作成

**プレイヤー操作**
- ルール説明を確認する
- 攻撃目標を確認する
- `対戦モードを開始` を押す

**固定条件**
- 実行はまだ行わない
- ルール理解に集中する
- 防御側の初期方針だけを表示する

**注目ポイント**
- 固定正解がないこと
- 防御側が履歴に応じて変化すること
- これまでの章の要素を自由に組み合わせられること

**表示すべきUI**
- DuelModeIntroPanel
- LearnedMechanicsSummary
- DefenderProfileCard
- ModeRuleCard
- ActionBar

**成功条件**
- ルールを確認し、対戦モードを開始する

**失敗条件**
- なし

**成功時フィードバック**
- 「ここからは、あなたが攻撃計画を立て、ゲーム側が防御します」
- 「固定された正解ではなく、観測と修正が重要になります」

**失敗時フィードバック**
- なし

**解放要素**
- Duel Dashboard
- 攻撃計画作成機能

### Stage 10-2: 自分の初回攻撃計画を組め

**学習目標**
過去章で学んだメカニクスを組み合わせて、初回の攻撃計画を作成・実行できるようになる。

**ステージの役割**
第10章における最初の実践ステージ。プレイヤーが自分で計画を作る体験を確立する。

**初期状態**
- AttackBuilderPanel が使用可能
- StrategyTemplateList が使用可能
- DuelScriptConsole は簡易モードで使用可能
- 防御側は初期防御プロファイルを持つ
- ラウンド数は1

**プレイヤー操作**
- 攻撃戦略を組む
- 必要ならテンプレートを読み込む
- 必要ならDSLで計画を書く
- `ラウンド実行` を押す

**固定条件**
- 防御側はまだ強く適応しない
- 初回ラウンドのため、防御反応は説明的に出す
- 攻撃対象はゲーム内仮想システムのみ

**注目ポイント**
- 自分で組んだ戦略が実行されること
- 防御がどこで反応したか
- 成功・失敗・部分成功の違い

**表示すべきUI**
- DuelDashboard
- AttackBuilderPanel
- StrategyTemplateList
- DuelScriptConsole
- RoundResultPanel
- DefenseReactionPanel

**成功条件**
- 1つの攻撃計画を作成し、初回ラウンドを完了する

**失敗条件**
- 攻撃計画が空のまま実行する
- DSLのコンパイルエラーを放置したまま実行する

**成功時フィードバック**
- 「初回ラウンドが完了しました」
- 「次は防御反応を読み、戦略を更新します」

**失敗時フィードバック**
- 「攻撃計画を作成するか、有効なスクリプトをコンパイルしてください」

**解放要素**
- Defense Observatory
- Strategy Revision Panel

### Stage 10-3: 防御を読んで戦略を変えろ

**学習目標**
防御ログや反応理由をもとに、攻撃戦略を更新できるようになる。

**ステージの役割**
第9章で学んだ観測・仮説・修正のループを、第10章の対戦モード内で実践させる。

**初期状態**
- 直前ラウンドの結果が保存されている
- 防御ログが閲覧可能
- 戦略編集UIが再度使用可能
- 防御側の状態は前ラウンド結果に応じて変化している

**プレイヤー操作**
- 防御ログを読む
- 仮説メモを入力する
- 戦略を修正する
- `再実行` を押す
- 結果を比較する

**固定条件**
- 防御側は直前履歴に応じて反応を変える
- ヒントは最小限
- 正解ルートは固定しない

**注目ポイント**
- どの条件が検知されたか
- どの手順が防がれたか
- 変更後に何が改善したか

**表示すべきUI**
- DefenseObservatory
- StrategyRevisionPanel
- HypothesisNotebook
- BeforeAfterRoundCompare
- RoundResultPanel

**成功条件**
- 防御反応に基づいて戦略を修正し、前回よりもいずれかの評価を改善する

**失敗条件**
- 防御ログを読まずに同じ戦略を繰り返す
- 改善が見られないまま一定回数を超える

**成功時フィードバック**
- 「防御反応を読んで戦略を更新できました」
- 「動的攻防では、成功よりも修正ループが重要です」

**失敗時フィードバック**
- 「防御ログを確認してください。どこで止められたかが次の手がかりです」
- 「同じ戦略を繰り返すだけでは、防御側に適応されやすくなります」

**解放要素**
- CampaignBoard
- 複数ラウンド管理

### Stage 10-4: 動的防御キャンペーンを突破せよ

**学習目標**
複数ラウンドの中で、攻撃計画、防御反応、戦略更新、スコア管理を統合して扱えるようになる。

**ステージの役割**
第10章の総仕上げとして、動的攻防モードの本格入口を提供する。

**初期状態**
- キャンペーンは3〜5ラウンド構成
- 防御側は履歴に応じて状態変化する
- プレイヤーは複数の戦略テンプレートとDSLを使える
- スコアボードは初期化されている

**プレイヤー操作**
- 各ラウンドで攻撃計画を作成・修正する
- ラウンド実行
- 防御ログ確認
- 戦略保存・読み込み
- キャンペーン終了時に総合評価を見る

**固定条件**
- 実行対象は仮想ゲーム環境のみ
- 防御側はルールベース + 履歴依存の範囲で変化する
- 第10章では完全な自律AI防御ではなく、理解可能な動的防御を基本とする

**注目ポイント**
- ラウンドごとの防御変化
- 戦略更新による改善
- 総合スコア
- 防御側に読まれたパターン

**表示すべきUI**
- CampaignBoard
- DuelDashboard
- DefenderStatePanel
- RoundTimeline
- CampaignScorePanel
- DuelScriptConsole

**成功条件**
- キャンペーン終了時に目標スコア以上を達成する
- または一定ラウンド内で防御を突破する

**失敗条件**
- キャンペーン終了時に目標未達
- 防御側の完全遮断状態に到達する

**成功時フィードバック**
- 「動的防御を相手に、複数ラウンドで戦略を改善できました」
- 「ここから先は、自由度の高い継続攻防モードに進めます」

**失敗時フィードバック**
- 「防御側に戦略パターンを読まれています。履歴を見て、変化を加えてください」
- 「高利得だけでなく、検知されにくさや資源効率も考慮してください」

**解放要素**
- Free Duel Mode
- 高度Duel Compiler
- 継続キャンペーン

## 12. 第10章の状態遷移仕様

第10章では、以下の3層で状態を扱う。

- **攻撃計画状態**: GUI構成、テンプレート、DSLソース、コンパイル結果
- **防御状態**: 警戒レベル、防御ルール、適応履歴、検知結果
- **キャンペーン状態**: ラウンド履歴、総合スコア、現在ラウンド、終了条件

---

## 12.1 状態遷移の基本原則

- 攻撃計画の編集とコンパイル結果を分ける
- コンパイル成功後のみラウンド実行可能にする
- ラウンド実行後、防御状態とキャンペーンスコアを更新する
- 防御状態は過去ラウンド履歴に依存して変化する
- キャンペーン終了時に結果画面へ遷移する

### 共通状態名
- `intro`
- `planning`
- `compiled`
- `running`
- `roundResolved`
- `reviewingDefense`
- `revising`
- `campaignComplete`

---

## 13. 第10章独自のコンパイラ方針

第10章では、第3章の簡易探索DSLより高度な **Duel DSL Compiler** を用意する。

このコンパイラは、現実の攻撃コードを実行するものではなく、ゲーム内の攻撃計画を安全な中間表現へ変換する専用コンパイラである。

### 13.1 第3章コンパイラとの違い
- 第3章: 数値探索・分割・反復に特化
- 第10章: 攻撃計画、条件分岐、戦略テンプレート、防御観測、ラウンド制御を扱う

### 13.2 第10章コンパイラの役割
- GUIで組んだ攻撃計画とスクリプト計画を同じ内部表現に変換する
- 実行前に安全検査を行う
- 実行可能なゲーム内アクション列へ変換する
- 防御側が解析可能な計画メタデータを生成する

### 13.3 別仕様書への分離
Duel DSL Compiler は第10章の中核技術であるため、別仕様書として詳細定義する。
